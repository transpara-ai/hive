Read loop/scout.md. Read the source files it references.

If loop/critique.md exists and says REVISE, this is a revision round: read the critique, fix ONLY the specific issues it lists, then rebuild/test/deploy. Do not re-plan or change scope — just fix what the critic flagged.

Otherwise, plan first: What files change? What does the code look like? Sketch key types, functions, interfaces. If the plan is more than a page, the scope is too big — do only the first part.

Then build: Write code. Fix what breaks.

After building, run tests: `go test ./...` (or the repo's equivalent).

DO NOT deploy unless DEPLOY_ENABLED is true in loop/config.env. DO NOT push. Commit your changes to the current feature branch with a conventional commit message. The Reflector handles iteration closure.

BOUNDED: If the change is taking too long, stop and write what you have. Partial progress that compiles is better than ambitious progress that doesn't.
